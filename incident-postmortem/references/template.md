# Postmortem 完全テンプレート

`incident-postmortem` skill 本体から呼び出される、フルテンプレート。
本体の SKILL.md は概要・原則のみ載せ、実際の出力はここを参照する。

```markdown
# Postmortem: <インシデントの一行サマリー>

| 項目 | 内容 |
| :--- | :--- |
| インシデントID | INC-YYYY-MMDD-NN |
| 発生日時 | YYYY-MM-DD HH:MM JST |
| 検知日時 | YYYY-MM-DD HH:MM JST(検知遅延 X 分) |
| 復旧日時 | YYYY-MM-DD HH:MM JST |
| 影響時間 | X 分 |
| 重大度 | SEV1 / SEV2 / SEV3 / SEV4 |
| 著者 | <名前> |
| レビュアー | <名前> |
| ステータス | Draft / Reviewed / Closed |

## TL;DR
2-3文で全体サマリー。何が起き、何が原因で、どうしたかを最短で伝える。

## 影響範囲
- **ユーザー影響**: 例) チェックアウトAPIが39分間にわたって 5xx を返却。約12,000ユーザーが影響。
- **ビジネス影響**: 例) 推定 ¥X,XXX,XXX の機会損失。
- **データ影響**: データ損失/不整合なし(✓)/あり(詳細→...)
- **外部告知**: ステータスページ更新あり / なし

## タイムライン(JST)

| :--- | :--- | :--- |

| HH:MM | IC(指揮)を任命 | - |
| HH:MM | 原因を疑う仮説 | <担当者> |
| HH:MM | ロールバック開始 | <担当者> |
| HH:MM | ロールバック完了、エラー率低下 | - |
| HH:MM | エラー率が平常値に復帰、復旧宣言 | IC |
| HH:MM | ユーザー向け告知、ステータスページ更新 | PM |

## 根本原因分析(Five Whys)

### 1. なぜ X が起きたか?
直接原因(例: 新コードが NullPointerException を投げていた)

### 2. なぜ ... ?
1段深い原因(例: データ移行で NULL のまま残っていた)

### 3. なぜ ... ?
さらに深い原因(例: 移行スクリプトに WHERE 条件のバグ)

### 4. なぜ ... ?
プロセス・設計の不備(例: 移行スクリプトのテストが少数サンプル)

### 5. なぜ ... ?
組織・文化の根本(例: 本番相当データでのドライラン文化が無い)

→ **真の原因**: <仕組み・プロセスの不備をひと言で>

## 何がうまくいったか
- アラート発火→対応開始まで X 分(目標達成)
- ロールバック手順が事前文書化されていた
- IC 体制が機能した

## 何がうまくいかなかったか
- 検知に X 分かかった(閾値が緩すぎた)
- 移行スクリプトのテストが甘かった
- ステージング環境のデータが本番と乖離

## 運が良かったこと
- ピーク時間帯ではなかった
- データ書き込みは正常で、再試行で回復可能だった

## Lessons Learned
- データ移行は「件数の少ないサンプル」では本番のエッジケースを再現できない
- 防御的プログラミング(NULL安全)と DB 制約の両方が無いと脆い
- アラート閾値は「ノイズになるから」と緩めすぎていた
```
