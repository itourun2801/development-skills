# Incident Postmortem

障害事後分析を Blameless(個人非難なし)の原則で文書化する。

## 大原則

- **人を責めない、仕組みを責める。** 個人ミスはほぼ常にシステム/プロセスの不備の表面化。
- **責任追及ではなく学習が目的。** 同じことが二度起きない仕組みを作る。
- **隠さない・歪めない。** 都合の悪い事実こそ価値がある。

## 入力情報

- インシデント発生・検知・復旧の時刻と経緯
- 影響範囲(ユーザー数、機能、収益影響)
- 対応者・対応内容のログ(Slack履歴、コマンド履歴等)
- 監視グラフ、エラーログ

## テンプレート

```markdown
# Postmortem: <インシデントの一行サマリー>

| :--- | :--- |

| 検知日時 | 2026-05-17 14:31 JST(検知遅延 8分) |
| 復旧日時 | 2026-05-17 15:02 JST |
| 影響時間 | 39分 |
| 重大度 | SEV2 |
| 著者 | <名前> |

| ステータス | Draft / Reviewed / Closed |

## TL;DR
2-3文で全体サマリー。何が起き、何が原因で、どうしたかを最短で伝える。

## 影響範囲
- **ユーザー影響**: チェックアウトAPIが39分間にわたって 5xx を返却。約12,000ユーザーが影響。
- **ビジネス影響**: 推定 ¥X,XXX,XXX の機会損失。
- **データ影響**: データ損失/不整合なし(✓)/あり(詳細→...)
- **外部告知**: ステータスページ更新あり / なし

| 15:30 | ユーザー向け告知、ステータスページ更新 | PM |

## 根本原因分析(Five Whys)

### 1. なぜチェックアウトAPIが5xxを返したか?
v1.5.2 で導入された `OrderService.calculateTax()` が、税率が未設定の商品で NullPointerException を投げていた。

### 2. なぜ税率未設定の商品が存在したか?
データ移行時に一部商品の `tax_rate` カラムが NULL のままになっていた。

### 3. なぜ NULL のままだったか?
移行スクリプトに WHERE 条件のバグがあり、特定カテゴリの商品が対象外になっていた。

### 4. なぜ移行スクリプトのバグがレビューを通過したか?
移行スクリプトのテストが少数のサンプルデータのみで、エッジケースを網羅していなかった。

### 5. なぜテストが不十分でも問題が出なかったか?
本番相当データでのドライランを実施していなかった。ステージング環境のデータ件数が本番と乖離していた。

→ **真の原因**: 本番相当データでのリハーサル文化の欠落

## 何がうまくいったか
- アラートが発火してから対応開始まで2分(目標5分以内)を達成
- ロールバック手順が事前に文書化されており、迷いなく実行できた
- IC体制が機能した

## 何がうまくいかなかったか
- 検知に8分かかった(エラー率の閾値が緩すぎた)
- 移行スクリプトのテストが甘かった
- ステージング環境のデータが本番と乖離していた

## 運が良かったこと
- ピーク時間帯ではなかった(影響ユーザー数が抑えられた)
- データ書き込みは正常で、5xxを受けたユーザーは単に再試行できた

## 再発防止アクション

| # | アクション | 種別 | 担当 | 期限 | Issue |
| :--- | :--- | :--- | :--- | :--- | :--- |
| 1 | エラー率アラートの閾値を 5% → 1% に下げる | 検知改善 | Aさん | 5/24 | #501 |
| 2 | 移行スクリプトのテストガイドライン整備 | 予防 | Bさん | 5/31 | #502 |
| 3 | ステージング環境のデータを月次で本番から同期 | 予防 | SREチーム | 6/15 | #503 |
| 4 | `tax_rate` の NULL を許容しないDB制約追加 | 防御 | Cさん | 5/31 | #504 |
| 5 | `calculateTax` に null安全なフォールバック追加 | 防御 | Cさん | 5/24 | #505 |
| 6 | デプロイ後5分のエラー率監視を自動化 | 検知改善 | SREチーム | 6/30 | #506 |

(種別: 予防 / 検知改善 / 影響軽減 / 復旧時間短縮)

## Lessons Learned
- データ移行は「件数の少ないサンプル」では本番のエッジケースを再現できない
- 防御的プログラミング(NULL安全)とDB制約の両方が無いと脆い
- アラート閾値は「ノイズになるから」と緩めすぎていた
```

## 重大度の目安(プロジェクトで調整)

| 重大度 | 定義 | 例 |
| :--- | :--- | :--- |
| SEV1 | 全機能停止 / データ損失 / セキュリティ侵害 | サイト全体ダウン、PII漏洩 |
| SEV2 | 主要機能停止 / 多数ユーザー影響 | 決済不可、ログイン不可 |
| SEV3 | 一部機能停止 / 限定的影響 | 検索機能のみ低速 |
| SEV4 | 軽微 / ワークアラウンドあり | 一部画面のレイアウト崩れ |

## 原則

- **「人為的ミス」で止めない。** それは原因の表面。なぜそのミスが起きる仕組みだったかを掘る。
- **アクションは具体的・期限付き・担当付き。** 「気をつける」は無効。Issue化して追跡する。
- **数値で書く。** 「結構な影響」ではなく「12,000ユーザー、39分間」。
- **正直に書く。** 自分(チーム)が悪く見える事実こそ価値。
- **共有する。** 関係者だけでなく組織全体に共有して学びを横展開する。
- **「うまくいったこと」も必ず書く。** 良い慣習を強化するため、悪い面の指摘だけにしない。
