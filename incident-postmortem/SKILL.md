# Incident Postmortem

障害事後分析を Blameless(個人非難なし)の原則で文書化する。

## 大原則

- **人を責めない、仕組みを責める。** 個人ミスはほぼ常にシステム/プロセスの不備の表面化。
- **責任追及ではなく学習が目的。** 同じことが二度起きない仕組みを作る。
- **隠さない・歪めない。** 都合の悪い事実こそ価値がある。

## 入力情報

- インシデント発生・検知・復旧の時刻と経緯
- 影響範囲(ユーザー数、機能、収益影響)
- 対応者・対応内容のログ(Slack 履歴、コマンド履歴等)
- 監視グラフ、エラーログ

## 出力構成(概要)

以下のセクションを含む Markdown 文書を生成する。フルテンプレートは [references/template.md](references/template.md) を参照。

1. **メタ情報** — インシデントID、発生/検知/復旧時刻、SEV、ステータス
2. **TL;DR** — 2-3 文の全体サマリー
3. **影響範囲** — ユーザー / ビジネス / データ / 外部告知
4. **タイムライン** — JST で時系列、担当者付き
5. **根本原因分析(Five Whys)** — 表面原因から仕組みの不備まで掘る
6. **何がうまくいった / いかなかった / 運が良かった**
7. **再発防止アクション** — 種別(予防/検知改善/影響軽減/復旧時間短縮)、担当、期限、Issue
8. **Lessons Learned** — 横展開すべき学び

## 重大度の目安

| 重大度 | 定義 | 例 |
| :--- | :--- | :--- |
| SEV1 | 全機能停止 / データ損失 / セキュリティ侵害 | サイト全体ダウン、PII 漏洩 |
| SEV2 | 主要機能停止 / 多数ユーザー影響 | 決済不可、ログイン不可 |
| SEV3 | 一部機能停止 / 限定的影響 | 検索機能のみ低速 |
| SEV4 | 軽微 / ワークアラウンドあり | 一部画面のレイアウト崩れ |

## 原則

- **「人為的ミス」で止めない。** それは原因の表面。なぜそのミスが起きる仕組みだったかを掘る。
- **アクションは具体的・期限付き・担当付き。** 「気をつける」は無効。Issue化して追跡する。
- **数値で書く。** 「結構な影響」ではなく「12,000ユーザー、39分間」。
- **正直に書く。** 自分(チーム)が悪く見える事実こそ価値。
- **共有する。** 関係者だけでなく組織全体に共有して学びを横展開する。
- **「うまくいったこと」も必ず書く。** 良い慣習を強化するため、悪い面の指摘だけにしない。
