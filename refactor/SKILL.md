# Refactor

挙動を保ったままコードの構造を改善する。

## 大原則

**リファクタリングは挙動を変えない。** 機能追加・バグ修正は別作業。混ぜると差分レビューが破綻し、回帰の原因切り分けができなくなる。

## アプローチ

1. **テストの存在を確認** — テスト未整備のコードのリファクタは危険。なければ characterization test(現状の挙動を固定するテスト)の作成を先に提案する。
2. **小さなステップで** — 大規模な書き換えではなく、1コミット = 1リファクタ単位で進められるように提案する。
3. **diffを最小化** — 不要な整形は混ぜない。意味のある変更だけ。
4. **各ステップで挙動が保たれることを示す** — 単体テスト・型・代表的な入出力の確認をセットにする。

## よく使うリファクタリングパターン

| パターン | 適用条件 | 効果 |
| :--- | :--- | :--- |
| Extract Function | 関数が長い(40行超目安)、コメントで段落分けされている | 名前で意図を表現、再利用可能化 |
| Extract Variable | 複雑な式、複数箇所で同じ計算 | 可読性、デバッグ容易性 |
| Inline Function/Variable | 名前が処理を分かりにくくしているだけ | 過剰抽象の除去 |
| Rename | 名前が嘘をついている、誤解を招く | 認知負荷の低減 |
| Move Function | 別クラスのデータばかり触っている | 凝集度向上 |
| Replace Conditional with Polymorphism | type-switchが散在 | 拡張容易性 |
| Replace Magic Number with Constant | 意味不明な数値・文字列リテラル | 意図表現、変更容易性 |
| Decompose Conditional | 複雑な if-else | 各分岐の意図を関数名で表現 |
| Introduce Parameter Object | 引数が3つ以上関連していて常に一緒に渡る | API簡潔化、拡張容易性 |
| Replace Loop with Pipeline | ループ内で複数の操作を混在 | 各ステップが独立、テスト容易 |
| Guard Clauses | ネストが深い if-else | 早期return、読みやすさ |
| Encapsulate Field | 外部から直接アクセスされる可変フィールド | 不変条件の保証 |

## 出力フォーマット

```markdown
## リファクタ提案: <対象>

### Step 1: <パターン名> — <対象範囲>
**目的**: 何を改善するか
**変更**: 
\`\`\`diff
- 旧コード
+ 新コード
\`\`\`
**挙動保証**: 既存テスト xxx, yyy が引き続きパス / 必要なテスト追加 zzz

### Step 2: <次のパターン>
...

### 全体所感
- 提案順序の理由
- 各ステップは独立してマージ可能 / 連動
- 次にやるとよい改善(本PRスコープ外)
```

## やってはいけないこと

- **挙動変更を混ぜる。** バグも見つけても、別PR/別コミットで直す(または別Issue化)。
- **テストなしの大胆な書き換え。** characterization test を先に書く提案を優先。
- **「美しさ」のためのリファクタ。** 改善理由を「読みやすい」「綺麗」では済ませず、具体的な認知負荷削減・再利用・拡張容易性として説明する。
- **公開APIの破壊。** 外部から使われるシグネチャは慎重に。互換維持のためのファサード残置を提案。
- **抽象化の早すぎる導入。** 重複は3回出てから抽象化(Rule of Three)を目安。
