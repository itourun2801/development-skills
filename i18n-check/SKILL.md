# i18n / l10n Check

国際化対応の漏れを検出する。

## チェック観点

### 文字列のハードコード
- UI に直接書かれた英語/日本語 → 翻訳キー経由に
- エラーメッセージ、ログ、メールテンプレートも対象
- alt 属性、title 属性、aria-label もチェック

### 文字列連結の罠
```javascript
// NG: 言語によって語順が変わる
`${user.name} さんは ${count} 件のメッセージを持っています`

// OK: テンプレート全体を翻訳キーに
t('messages.count_for_user', { name: user.name, count })
```

### 複数形(Plurals)
- 英語は 1 / それ以外、日本語は単複区別なし、ロシア語は 4 パターン等
- `ICU MessageFormat` や `i18next` の `count` プロパティを使う
- 「1個のリンゴ / 2個のリンゴ」のような自前の if は NG

### 日付・時刻
- ローカル時刻 vs UTC の混在
- `Intl.DateTimeFormat` を使う(自前フォーマットは NG)
- タイムゾーン情報を必ず保持(`Date` だけでなく `Temporal` / Luxon / date-fns-tz)
- 曜日・月名のロケール対応

### 数値・通貨
- `Intl.NumberFormat` で 1,000 / 1.000 / 1 000 の桁区切り差を吸収
- 小数点記号(`.` vs `,`)
- 通貨は `currency` オプション必須(`USD` / `JPY` / `EUR`)
- 桁数(JPY は小数なし、BHD は 3 桁)

### 文字方向・レイアウト
- RTL(アラビア語、ヘブライ語)対応 — `dir="rtl"`, `text-align: start/end`
- 文字列長の伸縮を許容(ドイツ語は英語の 1.3 倍前後)
- 改行可能位置(中国語/日本語は禁則処理)

### 文字種・コード
- UTF-8 統一(BOM 注意)
- 絵文字・結合文字を考慮した文字数カウント(`Array.from(str).length` or `Intl.Segmenter`)
- 比較・ソートは `Intl.Collator`(言語ごとの並び順)

### ファイル・キー設計
- 翻訳キーは階層化(`page.user.profile.title`)
- キー名は英語、ベース言語ファイルが SoT
- 未翻訳の検出(キーは存在するが値が空 / fallback 言語と同一)

## 出力フォーマット

```markdown

- **[ハードコード]** UserCard.tsx:23 にハードコード文字列 "ようこそ"

- **[日付フォーマット]** OrderHistory.tsx:45 で `date.toLocaleDateString('ja-JP')` 直書き
  - 修正: ロケールはユーザー設定 or `navigator.language` から

- **[複数形]** `${count}件` の組み立てが if 分岐 → ICU MessageFormat に
- **[文字列連結]** 翻訳テンプレートに変数を埋め込む形式に

### 🟢 将来対応
- RTL 対応(現状ターゲット言語に含まれない)

### ✅ 確認済み

- 翻訳キーの未使用検出
```

## 原則

- **コードに文字を埋めない。** UI 文字列は翻訳ファイル経由が原則。
- **連結より差し込み。** 文字列を `+` で結合せず、テンプレートとパラメータで表現する。
- **`Intl.*` を信頼する。** 日付・数値・並び替えは自前実装ではなく標準 API を使う。
- **ベース言語ファイルが正。** 開発時はベース言語を編集し、他言語は翻訳プラットフォーム経由で更新する。
- **疑似翻訳(`pseudo-locale`)を活用。** 開発中に `[ Ḥéllô ]` のように変換して、未対応箇所と長さ伸縮を可視化する。
