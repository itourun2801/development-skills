# Security Check

OWASP Top 10 / CWE Top 25 観点で、コード・設定・アーキテクチャの脆弱性を体系的に洗い出す。

## チェック観点

### A01: アクセス制御の不備
- 認可ロジックの欠落・バイパス、IDOR(他人のID指定でアクセス可)、デフォルトでpublicな設定

### A02: 暗号化の失敗
- 機密データの平文保存、弱いハッシュ(MD5/SHA1)、ハードコードされた秘密鍵、HTTPS非強制

### A03: インジェクション
- SQL/NoSQL/コマンド/LDAP/XPath/テンプレートインジェクション、未エスケープ出力(XSS)

### A04: 安全でない設計
- レート制限なし、ビジネスロジックの悪用余地、信頼境界の曖昧さ

### A05: セキュリティ設定ミス
- デバッグモード本番有効、デフォルト認証情報、不要なポート・機能、詳細エラー露出

### A06: 脆弱で古いコンポーネント
- 依存ライブラリのCVE、サポート切れランタイム

### A07: 識別・認証の失敗
- 弱いパスワードポリシー、セッション固定、トークン無期限、MFA未対応、ブルートフォース対策なし

### A08: ソフトウェアとデータ整合性の失敗
- 署名検証なしのアップデート、安全でないデシリアライズ、信頼できないCDN

### A09: ロギング・監視の不備
- 認証失敗・権限変更が記録されない、ログに機密情報が混入

### A10: SSRF
- ユーザー入力をそのままURLとしてfetch/curl、内部メタデータエンドポイントへの到達可能性

## 出力フォーマット

```markdown
## セキュリティ診断結果

### 🚨 Critical (即時対応)
- **[CWE-XX / OWASP A0X]** 問題の要約
  - 影響: どんな攻撃が成立するか
  - 該当箇所: ファイル:行
  - 修正方針: 具体的な対策

### ⚠️ High / Medium / Low
(同フォーマット)

### ✅ 確認済み(問題なし)
- チェックして問題なかった項目を明示

### 推奨される追加対策
- ライブラリ追加、設定変更、運用ルール
```

## 原則

- **過剰検出を恐れない。** セキュリティは見落としコストが致命的。疑わしきは指摘する。
- **CWE / OWASPの番号を必ず添える。** 受け手が深堀りできる足がかりになる。
- **「とは言え本番では発火しない理由」も書く。** 多層防御の前提で、現状のリスクと潜在リスクを切り分ける。
- **脆弱性の実証コードは出さない。** 修正のための情報に留め、攻撃手順そのものは書かない。
