# Rollback Guide

障害発生時の切り戻し手順を、コード・DB・設定・データ整合性まで含めて生成する。

## 入力情報

開始時に確認:

- **戻したいバージョン** — タグ/コミット/リリース番号
- **デプロイ方式** — Blue/Green / Rolling / Canary / Container / Serverless / 物理サーバ
- **DB変更の有無** — マイグレーションがあるか、データ変換を伴うか
- **影響範囲** — 一部リージョン / 全体 / 特定機能のみ
- **緊急度** — 即時(数分) / 計画的(数十分許容)

## ロールバック判断基準

事前に「何があったら戻すか」を決める:

| 指標 | しきい値例 |
| :--- | :--- |
| エラー率 | 平常時の3倍超が5分継続 |
| レイテンシ | p95 が SLO の2倍超 |
| ヘルスチェック | 連続失敗 |
| ビジネスKPI | 注文数が急減、決済成功率低下 |
| 顧客報告 | 重大な機能不全の報告複数 |

## ロールバック手順テンプレート

```markdown
## ロールバック手順: <リリース名 / 日時>

### 0. 状況確認(2分以内)
- [ ] 障害事象をSlack `#incident` に第一報
- [ ] インシデント指揮者(IC)を決定
- [ ] 影響範囲を初期見積もり

### 1. Go/No-Go判断
- [ ] ロールバックが本当に解決するか確認(設定だけの問題ならフィーチャーフラグでOFFが早い)
- [ ] ロールバックでさらなる問題が起きないか確認(DBスキーマ依存等)
- [ ] 関係者に意思決定を共有

### 2. フィーチャーフラグでの無効化(可能なら最優先)
\`\`\`bash
# 該当機能を即時OFF
launchdarkly flag set new-checkout-flow --value false
\`\`\`
- 効果: 数秒〜数分でユーザー影響を止められる
- 注意: 一部のセッションで状態不整合が起きる可能性

### 3. コードのロールバック
**Blue/Greenの場合:**
\`\`\`bash
# トラフィックを旧スタックに切り戻し
aws elbv2 modify-listener --listener-arn ... --default-actions ...
\`\`\`
- 確認: ヘルスチェック緑、エラー率正常化

**Rollingの場合:**
\`\`\`bash
# 直前リビジョンに戻す
kubectl rollout undo deployment/api -n prod
kubectl rollout status deployment/api -n prod
\`\`\`

**Container Registry の場合:**
\`\`\`bash
# v1.3.0 → v1.2.5 にタグ差し替え + 再デプロイ
\`\`\`

### 4. DBマイグレーションの扱い
**Case A: 後方互換のあるマイグレーション(カラム追加等)**
- ロールバック不要。新コードは旧スキーマで動く前提で設計されている。

**Case B: 後方非互換(カラム削除/型変更等)**
- ⚠️ 単純なdownでデータ消失リスクあり
- 検討事項:
  - down前にデータをバックアップ(`pg_dump`等)
  - データ移行スクリプトの逆処理が必要か
  - downを実施せず、コードだけ戻して旧スキーマと互換になるよう一時パッチを当てる選択肢
- 実行(慎重に):
\`\`\`bash
bundle exec rails db:rollback STEP=1
# または
npx prisma migrate resolve --rolled-back <migration_name>
\`\`\`

### 5. 設定・環境変数
- [ ] 新規追加した環境変数を旧値に戻す(または削除)
- [ ] 外部サービス側の設定(Webhook URL/コールバック)を戻す
- [ ] CDN/キャッシュをパージ

### 6. データ整合性の回復
- [ ] 新バージョンで作成された不正データを抽出
\`\`\`sql
SELECT * FROM orders WHERE created_at > 'YYYY-MM-DD HH:MM' AND status = 'invalid_xxx';
\`\`\`
- [ ] 修復方針を決定(削除 / 修正 / 補償処理)
- [ ] 対象顧客への連絡が必要か判断

### 7. 確認
- [ ] エラー率が平常値に戻った
- [ ] レイテンシが平常値
- [ ] 主要機能のスモークテスト
- [ ] 監視ダッシュボードで5-10分間安定確認

### 8. 事後
- [ ] インシデント解決の連絡
- [ ] ポストモーテム会のスケジュール設定
- [ ] 影響を受けたユーザーへの対応(必要時)
- [ ] CHANGELOG にロールバック記録
```

## 原則

- **時間との戦い。** ロールバック判断を引っ張らない。「もう少し様子を見る」が傷を広げる。
- **フィーチャーフラグを最優先。** 一番速く、影響範囲も限定的。設計段階で仕込んでおく価値がある。
- **DBダウンマイグレーションは慎重。** 「コードを戻すだけ」で済む設計(後方互換マイグレーション)が本来理想。
- **手順は事前にリハーサル。** 本番初遭遇は事故の元。ステージングで実際にロールバックしておく。
- **記録を残す。** いつ何を実行したかを Slack / ログに時系列で残す。ポストモーテムで必須。
