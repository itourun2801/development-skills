# Rollback Guide

障害発生時の切り戻し手順を、コード・DB・設定・データ整合性まで含めて生成する。

## 入力情報

開始時に確認:

- **戻したいバージョン** — タグ/コミット/リリース番号
- **デプロイ方式** — Blue/Green / Rolling / Canary / Container / Serverless / 物理サーバ
- **DB 変更の有無** — マイグレーションがあるか、データ変換を伴うか
- **影響範囲** — 一部リージョン / 全体 / 特定機能のみ
- **緊急度** — 即時(数分) / 計画的(数十分許容)

## ロールバック判断の基本

事前に「何があったら戻すか」を決めておくのが望ましい。エラー率・レイテンシ・ヘルスチェック・ビジネス KPI・顧客報告のしきい値を組み合わせる。

具体的なしきい値例とフル手順テンプレートは [references/full-procedure.md](references/full-procedure.md) を参照。

## 手順の流れ(概要)

実際の出力は番号付きチェックリスト形式で。フル手順は references 参照。

1. **状況確認** — Slack 第一報、IC 任命、影響範囲見積もり
2. **Go/No-Go 判断** — 本当にロールバックが解決するかの確認
3. **フィーチャーフラグでの無効化** — 可能なら最優先(数秒〜数分で影響停止)
4. **コードのロールバック** — Blue/Green / Rolling / Container それぞれの手順
5. **DB マイグレーションの扱い** — 後方互換ならスキップ、非互換は要慎重
6. **設定・環境変数の復旧**
7. **データ整合性の回復** — 新バージョンで作成された不正データの抽出と対応
8. **確認** — エラー率・レイテンシ・スモークテスト
9. **事後** — ポストモーテム手配、ユーザー対応、CHANGELOG 記録

## 原則

- **時間との戦い。** ロールバック判断を引っ張らない。「もう少し様子を見る」が傷を広げる。
- **フィーチャーフラグを最優先。** 一番速く、影響範囲も限定的。設計段階で仕込んでおく価値がある。
- **DB ダウンマイグレーションは慎重。** 「コードを戻すだけ」で済む設計(後方互換マイグレーション)が本来理想。
- **手順は事前にリハーサル。** 本番初遭遇は事故の元。ステージングで実際にロールバックしておく。
- **記録を残す。** いつ何を実行したかを Slack / ログに時系列で残す。ポストモーテムで必須。
