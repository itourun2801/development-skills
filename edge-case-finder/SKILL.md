# Edge Case Finder

対象のコード・API・フォーム・仕様に対し、見落としやすい境界値・異常系・例外パターンを系統的に列挙する。

## 観点チェックリスト

### 値の境界
- **数値**: 0, 1, -1, 最大値(`Number.MAX_SAFE_INTEGER` / `INT_MAX`), 最小値, 桁あふれ, 浮動小数の誤差(`0.1 + 0.2`), 負の0, NaN, Infinity
- **文字列**: 空文字 `""`, 空白のみ `"   "`, 巨大長(1万文字), 改行混入, タブ, ゼロ幅文字, 絵文字, 結合文字(é vs é), RTL文字, NULL文字 `\0`
- **配列/コレクション**: 空, 1要素, N=最大件数, 重複要素, 順序逆転, ネスト深さ
- **日付/時刻**: うるう年(2/29), うるう秒, タイムゾーン跨ぎ, サマータイム切替, 1970年以前, 9999年, 月末(2/28-2/30, 4/30 vs 4/31)
- **null/undefined**: 各引数で `null` `undefined` を入れたとき
- **型の境界**: 数値が文字列で来る("42"), booleanが数値(0/1), 配列がオブジェクト

### 入力の悪意・破損
- 想定外の型, 想定外のキー追加, 必須キー欠落, JSONとして不正
- 文字コード(UTF-8以外, BOM付き, 不正バイト列)
- 巨大ペイロード(MB/GB級)
- SQLインジェクション / XSS / コマンドインジェクション パターン

### 環境・実行時
- **並行性**: 同時実行で競合, 同一リクエストの二重送信, ロック順序
- **冪等性**: 同じリクエストを2回送って2回処理されないか
- **リトライ**: ネットワーク切断中の再試行, 一部成功状態からの再開
- **ロケール**: 日付フォーマット, 通貨, 小数点記号(`,` vs `.`)
- **時刻依存**: 真夜中跨ぎ, 月末跨ぎ, 年末跨ぎ
- **乱数依存**: 確率0%/100%パスの存在

### ビジネスロジック
- **境界条件**: 「以上/超」「以下/未満」のオフバイワン
- **状態遷移**: 想定外の状態から呼ばれたとき(削除済みリソースへの操作等)
- **権限**: 自分のリソース vs 他人のリソース, 役割境界
- **金額・在庫**: マイナス, 0, 最大値, 通貨混在, 端数処理

### 外部依存
- DB: コネクション枯渇, タイムアウト, デッドロック, レプリカ遅延
- API: 503/504, タイムアウト, レスポンス遅延, 不正レスポンス, ステータスとボディの不整合
- ファイル: 存在しない, 権限なし, ロック中, シンボリックリンク, 巨大ファイル

## 出力フォーマット

```markdown
## エッジケース一覧: <対象名>

### 🔴 高優先(本番で起きうる)
- **[境界値] 数量0での購入**
  - 入力: `quantity = 0`
  - 想定動作: バリデーションエラー
  - 現状: 未確認 → テスト追加推奨

### 🟡 中優先
- **[並行性] 同一ユーザーの二重送信**
  - シナリオ: ネットワーク遅延でユーザーが2回送信ボタンを押す
  - 想定: 1回だけ処理
  - 対策案: idempotency-key, DB一意制約

### 🟢 低優先(発生確率は低いが理論上ある)
- ...

### ✅ カバー済み確認
- 既存テスト/コードで明示的に処理されているケース
```

## 原則

- **質より網羅。** リストアップ段階では「あり得るか?」より「あり得るとして考慮したか?」を優先。
- **優先度を必ず付ける。** 全部対応はコスト過大。発生頻度 × 影響度で並べる。
- **「想定動作」と「現状」を分ける。** どう振る舞うべきかと、実際にどう振る舞うかは別問題。
- **テスト化を促す。** 列挙したケースは可能なら `test-gen` に流す前提で書く。
