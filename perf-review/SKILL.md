# Performance Review

パフォーマンス問題を体系的に洗い出す。計測なしの推測最適化は避けつつ、明らかなアンチパターンは指摘する。

## チェック観点

### データベース
- **N+1クエリ** — ループ内でクエリ発行していないか
- **不要なSELECT \*** — 必要カラムだけ取得しているか
- **INDEX不足** — WHERE/JOIN/ORDER BYで使われるカラムにINDEXがあるか
- **大量データのオンメモリ処理** — ページング/ストリーミング/バッチ化されているか
- **トランザクション範囲** — 必要以上に長く保持されていないか
- **デッドロック誘発** — ロック順序、行ロック範囲

### アプリケーションロジック
- **不要なループ** — 二重ループ、O(n²)以上の処理
- **同期処理の直列化** — 並列化可能なI/Oが直列実行されていないか
- **重い処理のキャッシュ漏れ** — 同じ計算/取得を繰り返していないか
- **メモリリーク** — クロージャ参照、リスナー解除漏れ、グローバル蓄積

### ネットワーク
- **過剰なAPI呼び出し** — まとめられるリクエスト
- **大きなペイロード** — 圧縮、不要フィールド除去
- **CDN/キャッシュ未活用** — 静的アセット、HTTPキャッシュヘッダ

### フロントエンド
- **不要な再レンダリング** — memo化、key適切性、状態配置
- **バンドルサイズ** — 動的import、tree shaking
- **画像最適化** — 適切なフォーマット・サイズ・lazy load

## 出力フォーマット

```markdown
## パフォーマンス診断結果

### 🔴 Critical (本番で問題になる)
- **[N+1問題]** UserController.list
  - 該当: 行 45-52
  - 現状: ユーザー数N件ごとに `posts` を発行 → 合計 N+1 クエリ
  - 改善: `includes(:posts)` または JOIN で1クエリにまとめる
  - 期待効果: 1000ユーザーで ~1000ms → ~50ms

### 🟡 Should Improve
(同フォーマット、定量見積もりがあれば併記)

### 🟢 Future Optimization
- 今すぐ困らないが、規模拡大時に問題化する箇所

### 計測の推奨
- 推測で直す前に計測すべき箇所(プロファイラ、APM、EXPLAIN等)
```

## 原則

- **計測ファースト。** 「ここが遅いはず」だけで直さない。可能なら EXPLAIN / プロファイラ / ベンチを推奨する。
- **改善効果を見積もる。** 「N+1で1000ms → JOIN化で50ms想定」のように、オーダーで示す。
- **可読性とのトレードオフを明示する。** 最適化で複雑になる場合は、影響度と引き換えに値するかを述べる。
- **早すぎる最適化は指摘しない。** ホットパスでない箇所のマイクロ最適化は Future 扱い。
