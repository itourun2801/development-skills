# Pull Request Description

変更内容からPR説明文を生成する。

## 入力情報

開始時に把握する:

- **変更内容** — `git log` / `git diff` / 説明
- **関連Issue/Ticket** — Jira / GitHub Issue
- **PRテンプレート** — `.github/pull_request_template.md` があれば優先採用
- **対象ブランチ** — main / develop / release/* 等

## 構成テンプレート(汎用)

```markdown
## 概要 / Summary
1〜2文で「何を、なぜ」変更したか

## 背景 / Context
- なぜこの変更が必要か
- 関連Issue: #123
- 関連ドキュメント: <link>

## 変更内容 / Changes
- [機能] ユーザー登録時のメール確認フローを追加
- [機能] `/auth/verify` エンドポイント新設
- [改修] 既存の `/auth/register` レスポンスに `verification_required` フィールドを追加
- [テスト] ユニットテスト 12件、E2Eシナリオ 3件追加

## スクリーンショット / Screenshots
| Before | After |
| :--- | :--- |
| (UI変更がある場合) | |

## 動作確認 / How to Test
1. `pnpm install` 後 `pnpm dev`
2. `/register` でアカウント作成
3. 受信メールのリンクを踏み、ダッシュボードに遷移することを確認

## 影響範囲 / Impact
- **ユーザー影響**: 新規登録フローが変わる(既存ユーザーは影響なし)
- **DB**: マイグレーション 1件追加(後方互換あり)
- **環境変数**: `MAIL_FROM_ADDRESS` 追加が必要
- **API互換性**: 既存クライアントは動作継続(新フィールドは optional)

## レビュー観点 / Review Notes
- ⚠️ メール送信のエラー時の挙動を特に見てほしい(`services/mail.ts:80`)
- 暗号化方式の選定理由は `docs/decisions/0012.md` 参照

## チェックリスト
- [ ] テスト追加・更新済み
- [ ] ドキュメント更新済み(README/API spec)
- [ ] CHANGELOG 更新済み
- [ ] 環境変数の追加を共有済み
- [ ] 破壊的変更なし(または移行手順を記載)
- [ ] セルフレビュー実施済み

Closes: #123
```

## バリエーション

### Bugfix PR

```markdown
## 概要
〜のバグを修正。

## 再現手順
1. ...
2. ...

## 原因
〜が〜の条件で〜してしまっていた。

## 修正方針
〜することで対応。代替案として〜も検討したが、〜の理由で採用しなかった。

## 影響範囲・回帰リスク
- 影響: ...
- 追加テスト: 該当ケースの回帰テストを `xxx.test.ts` に追加

Fixes: #234
```

### Refactor PR

```markdown
## 概要
挙動を変えずに〜を整理。

## 動機
- 〜が分散していて変更コストが高い
- 今後の機能追加(#300)に備える前準備

## 変更内容
- ...
- ⚠️ 公開API/外部仕様は変更していない

## 挙動が変わっていないことの確認
- 既存テスト全件パス
- カバレッジ低下なし(80% → 81%)
- 主要画面のスモークテスト実施
```

### Infra PR

```markdown
## 概要
〜の設定を変更。

## 変更内容
- ...

## ロールバック手順
- ...

## 監視・確認項目
- メトリクス: ...
- ログ: ...
```

## 原則

- **「なぜ」を最優先で書く。** 「何を変えたか」はdiffから読める。動機・意図・選択しなかった代替案を書く。
- **レビュアー目線で書く。** どこを重点的に見るべきか、どの順序で読むと理解しやすいかを案内する。
- **動作確認手順は再現可能に。** 「動きました」だけでなく、レビュアーが手元で追える手順。
- **影響範囲を正直に書く。** 「影響なし」と書くなら、その根拠もセットで。
- **既存テンプレートを尊重。** `.github/pull_request_template.md` があればそのフォーマットを採用。
- **絵文字で章を区切ると視認性UP。** ただしプロジェクト文化に合わせる。
