# Task Breakdown

大きな要件を、独立して実装・レビュー・マージできるサブタスクに分解する。

## 入力情報

開始時に把握する(不明なら質問):

- **元タスク** — 機能要件、ユーザーストーリー、依頼内容
- **粒度の目安** — 1サブタスクの想定時間(半日 / 1日 / 数日)
- **チーム構成** — 並列実行可能な人数、得意領域
- **既存制約** — 期限、ブロック要因、依存システム

## 分解の方針

### INVEST原則
- **Independent** — 他のタスクに依存しない、または依存が明示されている
- **Negotiable** — 詳細は実装時に調整可能(過度な事前定義をしない)
- **Valuable** — 単独でも何らかの価値を生む(または親タスクの一部として明確に貢献)
- **Estimable** — 見積もりできる程度に内容が明確
- **Small** — 数時間〜数日で完了する大きさ
- **Testable** — 完了条件が客観的に検証可能

### 分解の切り口(複数組み合わせる)

1. **垂直スライス** — DB→API→UI を最小機能セットで一気通貫(MVP化)
2. **水平スライス** — レイヤー単位(DB変更 → API実装 → UI実装 → E2E)
3. **ユーザーフロー** — Happy Path → Alternative → Error Handling
4. **CRUD分割** — Read 先行 → Create → Update → Delete
5. **ロール別** — 管理者機能 → 一般ユーザー機能
6. **インフラ vs アプリ** — 環境構築 / マイグレーション / アプリ実装 / 監視追加
7. **必須 vs Nice to Have** — Phase 1(コア) / Phase 2(拡張)

垂直スライスを基本に、その内側を水平に切るのが多くの場合扱いやすい。

## サブタスクの「良い」見分け方

- 1人で2-3日で完了できる
- 単独でレビュー可能(他PRに依存しない、または依存が明示)
- 「完了」が客観的に判定できる受け入れ基準を持つ
- タイトルだけで何をするか8割伝わる

## 原則

- **垂直スライスを優先。** 「DB全部 → API全部 → UI全部」より「最小機能を一気通貫」のほうが価値が早く出る。
- **依存関係を明示。** どれが何の後にやるべきかを図で示す。並列実行可能なものを最大化する。
- **見積もりは桁感で十分。** 「0.5d / 1d / 2d / 3d+」程度の粒度。それより大きいものは更に分解。
- **Nice to Have は別Phase。** MVPに不要なものは混ぜない。スコープ膨張を防ぐ。
- **「何をしないか」を書く。** スコープ外を明示しないと、後から要望が混入する。
- **見積もり不能なものは「調査タスク」を切る。** いきなり実装タスクにしない。
