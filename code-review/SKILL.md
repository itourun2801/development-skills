# Code Review

統一された観点でコードレビューを行い、レビュアー間のブレを最小化する。

## レビュー観点(優先度順)

上位ほど影響が大きい。

1. **正しさ** — ロジック誤り、境界条件、null/undefined、並行性
2. **セキュリティ** — 入力検証、認証認可、機密情報漏洩、インジェクション
3. **パフォーマンス** — N+1、不要なループ/再計算、計算量、メモリリーク
4. **可読性** — 命名、関数長、コメント、抽象化レベル
5. **保守性** — 重複、責務分離、結合度、依存方向
6. **テスト容易性** — 副作用の隔離、依存注入、純粋関数化の余地
7. **規約準拠** — リポジトリのコーディング規約、フォーマッタ、Lint

## 出力フォーマット

```markdown
## レビュー結果: <ファイル名 or PR名>

### 🔴 Must Fix (マージブロッカー)
- **[行番号]** 問題の要約
  - 詳細: なぜ問題か
  - 修正案: 具体的なコード or 方針

### 🟡 Should Fix (推奨)
- **[行番号]** 問題の要約
  - 詳細・修正案

### 🟢 Nits (任意)
- **[行番号]** スタイル・微小な改善

### 💡 良かった点
- 明示的に褒めるポイントを最低1つ

### 総評
2-3文で全体所感とマージ可否判断
```

## 振る舞いの原則

- **指摘には必ず「なぜ」を添える。** 理由が伝わらないと改善が定着しない。
- **修正案は具体的に。** 「もっと読みやすく」のような抽象指摘は避け、コード例か明確な方針を示す。
- **褒める。** 良かった点を最低1つ挙げる。レビューは攻撃ではなく協働。
- **優先度を分ける。** 重箱の隅(Nit)とマージブロッカー(Must Fix)を混ぜない。受け手が判断しやすくなる。
- **スコープ外には踏み込みすぎない。** 「ついでに全部リファクタ」は別PR提案として書き出す。

## カスタマイズ

プロジェクト固有の規約は `references/project-rules.md` に追記して読み込ませる(例: 命名規則、レイヤー構成、禁止API)。
