# PII / Privacy Check

PII(個人識別情報)や機密データの不適切な取扱いを検出する。

## PII / 機密データの分類

| 機密度 | 例 |
| :--- | :--- |
| **直接識別子** | 氏名、メールアドレス、電話番号、マイナンバー、SSN、住所、生年月日 |
| **間接識別子** | IP アドレス、Cookie ID、端末 ID、ユーザーエージェント、位置情報 |
| **特別カテゴリ** | 健康情報、人種、政治信条、性的指向、犯罪歴、生体情報(GDPR Art.9) |
| **金融** | クレカ番号、銀行口座、取引履歴 |
| **認証** | パスワード、トークン、API キー、セッション ID |
| **内部機密** | 顧客リスト、未公開財務情報、ソースコード |

## チェック観点

### ログ
- パスワード、トークン、クレカ番号、PII を `console.log` / logger でそのまま出していないか
- リクエストボディ・ヘッダーのダンプに `Authorization` / `Cookie` が含まれていないか
- スタックトレースに変数の中身が出る場合の漏洩
- マスキングルールの実装と適用範囲

### エラー応答
- 内部エラーをそのまま外部に返していないか(stacktrace, SQL 文, ファイルパス)
- メールアドレス存在確認の挙動(タイミング攻撃で会員判別できないか)

### URL / 経路
- 機密情報を query string に載せていないか(URL は履歴・ログ・Referrer に残る)
- メール内のマジックリンクの有効期限・1回使い切り
- リダイレクト URL の検証(オープンリダイレクト + 機密情報持ち回り)

### 分析・トラッキング
- GA / Mixpanel / Amplitude などの分析イベントに PII を載せていないか
- ヒートマップ・セッション録画(Hotjar/FullStory)でフォーム入力をマスクしているか
- Sentry/Rollbar への送信時のスクラビング設定

### 外部送信
- third-party API 呼び出しで必要以上の情報を送っていないか
- AI/LLM API への PII 送信(プロンプトに含まれていないか)
- データ越境(EU データを US リージョンに送る等)

### 保存・バックアップ
- ローカルストレージ・IndexedDB に機密情報を平文保存していないか
- DB に保存される PII の暗号化(at-rest)
- バックアップ・スナップショットの暗号化と保持期間
- 退会後の削除フロー(完全削除 / 匿名化)

### 同意・権利対応
- Cookie 同意バナーの実装(必須/任意の分類)
- データ主体の権利対応(開示・訂正・削除・ポータビリティ)
- 利用目的・保存期間の明示

## 出力フォーマット

```markdown
## PII / プライバシー診断結果

### 🚨 Critical(即時対応)
- **[ログ漏洩]** `auth/login.ts:42` で `logger.info(req.body)` がパスワードをログに記録
  - 影響: パスワードがログ基盤に保存される
  - 修正: `req.body` から password を除外、または専用ロガーで scrub

- **[URL 漏洩]** パスワードリセットトークンが query string で渡されている
  - 影響: Referrer ヘッダー経由で外部サイトに漏洩
  - 修正: POST + body で渡す、または短期 hash トークン

### ⚠️ High
- **[分析]** GA に `user_email` がユーザーパラメータとして送信されている
  - 修正: ID は内部で hash 化したものを送る

- **[Sentry]** 例外通知時のリクエストボディにメールアドレスが含まれる
  - 修正: Sentry の `beforeSend` で PII を除外

### ✅ 確認済み
- パスワードは bcrypt でハッシュ化
- クレカ番号は保存していない(Stripe トークン化)

### 推奨される追加対策
- 退会フローのデータ削除手順の文書化
- LLM API への送信ログのレビュー
- Cookie 同意管理ツールの導入検討
```

## 原則

- **必要最小限の収集。** 「あったら便利」では集めない。利用目的に直結するものだけ。
- **ログには PII を載せない。** マスキングをデフォルトに、解除は明示的な許可制で。
- **URL は最も漏れやすい経路。** クエリパラメータでの機密情報受け渡しを禁止する。
- **第三者送信は最小化。** 分析・エラー監視・LLM などへの送信前にスクラビング必須。
- **削除可能性を最初から設計に組み込む。** 後付けで「全削除フロー」を作るのは極めて高コスト。
- **法的要件は地域差がある。** GDPR / 個人情報保護法 / CCPA など、対象ユーザーの所在で要件が変わる。
