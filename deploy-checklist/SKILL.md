# Deploy Checklist

本番デプロイ前のチェックリストを提示する。スキップしてもよい項目はプロジェクト性質に応じて取捨選択。

## 入力情報

開始時に以下を確認する(不明なら質問):

- **リリース内容** — 機能追加 / バグ修正 / インフラ変更 / 設定のみ
- **規模** — 影響範囲(全ユーザー / 一部 / 内部のみ)
- **ロールバック可否** — DBマイグレーションを伴うか
- **デプロイ方式** — Blue/Green / Rolling / Canary / In-place

## チェックリスト

### 📦 コード・ビルド
- [ ] 全テストがパス(ユニット・統合・E2E)
- [ ] Lint / Format / 型チェックがパス
- [ ] CIが緑
- [ ] ステージング環境で動作確認済み
- [ ] 削除すべき console.log / debugger / TODO を残していない
- [ ] フィーチャーフラグの初期値が意図通り
- [ ] バージョン番号を更新(package.json / VERSION 等)

### 🗄️ データベース
- [ ] マイグレーションが冪等(複数回実行で壊れない)
- [ ] ロールバック手順(downマイグレーション)が用意されている
- [ ] 大規模テーブルへの変更はロック影響を確認(オンラインDDL / pt-online-schema-change等)
- [ ] バックアップが直近で取得されている
- [ ] 後方互換性: 旧コードと新スキーマが共存可能か(逆も)
- [ ] データ移行スクリプトのドライラン実施

### 🔐 セキュリティ・コンプライアンス
- [ ] 機密情報(API key/パスワード)がコード/ログに残っていない
- [ ] 依存パッケージの脆弱性スキャン済み(`npm audit` / `pip-audit` 等)
- [ ] 認可ロジックの追加・変更があれば監査ログを残せているか
- [ ] PII を扱う変更ならプライバシーレビュー済み

### ⚙️ 環境・設定
- [ ] 新規環境変数が本番のシークレットマネージャに登録済み
- [ ] 既存環境変数の変更を反映済み
- [ ] 機能フラグ・A/Bテスト設定確認
- [ ] CDN / キャッシュのパージが必要か検討
- [ ] 外部サービス側の設定(Webhook URL等)も更新済み

### 📊 監視・可観測性
- [ ] 新機能/エンドポイントのメトリクス・ログを出している
- [ ] アラートが新しいエラー型をカバーしている
- [ ] ダッシュボードに必要なグラフを追加済み
- [ ] エラー監視(Sentry等)のリリースタグを設定

### 🚀 デプロイ手順
- [ ] デプロイ手順書を最新化
- [ ] ロールバック手順を整理(下記 `rollback-guide` 参照)
- [ ] 担当者・連絡先(オンコール)を明確化
- [ ] デプロイウィンドウ(時間帯)が妥当(金曜夕方は避ける等)
- [ ] 想定所要時間とサービス影響の把握

### 📣 コミュニケーション
- [ ] 関係者(PM/CS/サポート/営業)に変更内容を共有
- [ ] CHANGELOG / リリースノート更新
- [ ] ユーザー向け告知が必要なら準備済み
- [ ] ステータスページの予約メンテ告知(必要時)

### ✅ デプロイ直後の確認(post-deploy)
- [ ] ヘルスチェック緑
- [ ] エラー率が平常値
- [ ] レイテンシが平常値
- [ ] 主要なユースケースの動作確認(スモークテスト)
- [ ] ログにマイグレーション/起動の正常完了が記録されている

## 出力フォーマット

上記をプロジェクトに合わせて取捨選択し、Markdownチェックボックス形式で提示。
各項目に「なぜ重要か」「スキップしてよい条件」を簡潔に添えると、形骸化を防げる。

## 原則

- **「全部チェック」を強要しない。** 当該リリースに関係ない項目は明確に除外する。
- **Go/No-Go判断者を明示。** チェックリスト埋めただけで安心しない。
- **過去の障害から学ぶ。** プロジェクトで起きた過去の障害を再発防止項目として追記する余地を残す。
- **ロールバック前提で考える。** 「戻せない」変更を最大限避ける設計を促す。
